### Раздел 1. Подготовка и генерация данных

#### 1.1 Настройка окружения Java

Первоначально на моей ОС (Fedora 43) была установлена Java 25. Но для корректной работы генератора Synthea требуется Java 21 (LTS).

```bash
# Установка JDK 21 (инструменты разработки)
sudo dnf install java-21-openjdk-devel -y

# Переключение версии Java (выберите номер, соответствующий java-21-openjdk)
sudo alternatives --config java
sudo alternatives --config javac

# Проверка версии
java -version  # Ожидается: openjdk version "21.x.x"
```

#### 1.2 Установка и сборка Synthea

Инструмент устанавливается в директорию `tools/` проекта.

```bash
mkdir -p tools && cd tools
git clone https://github.com/synthetichealth/synthea.git
cd synthea
./gradlew build -x test
```

#### 1.3 Конфигурирование параметров экспорта (FHIR R4)

Отредактируйте файл `tools/synthea/src/main/resources/synthea.properties` для соответствия спецификации проекта:

```properties
# Путь: tools/synthea/src/main/resources/synthea.properties
exporter.base_version = R4
exporter.fhir.export = true
exporter.fhir.transaction_bundle = true
exporter.ccda.export = false
exporter.csv.export = false
exporter.text.export = false
```

#### 1.4 Генерация синтетической популяции

Запуск оркестрации генерации через Python-скрипт. Скрипт настроен на возрастную группу 50-90 лет для достижения необходимой превалентности рака толстой кишки.

```bash
# Из корня проекта:
python3 src/synthea_manager.py
```

**Результат:** Данные в формате JSON сохраняются в `data/raw_synthea/fhir/`.

#### 1.5 Клинический аудит и профилирование данных

Проверка качества выборки и поиск диагностических меток (стадий) перед загрузкой в БД.

```bash
# Запуск инспектора (проверка структуры ресурсов и кодов SNOMED-CT)
python3 src/quick_check.py
```

### Раздел 2. Развертывание HAPI FHIR сервера

В связи с ограничениями доступа к публичным реестрам образов и необходимостью тонкой настройки FHIR-сервера, развертывание выполняется путем локальной сборки образа из исходных кодов.

#### 2.1 Системные требования

- **ОС:** Fedora Linux 43
- **CPU:** Intel Core i7 (рекомендуется 8+ потоков)
- **RAM:** 32 GiB (под нужды HAPI FHIR выделено 12 GiB)
- **Стек:** Docker Engine, Docker Compose, Python 3.12+, PostgreSQL 15.

#### 2.2 Подготовка окружения

Перед началом убедитесь, что Docker установлен и настроен на использование зеркал (Registry Mirrors) в `/etc/docker/daemon.json` для загрузки базовых образов (Java/Maven).

#### 2.3 Развертывание инфраструктуры

Для обеспечения модульности и контроля на каждом этапе развертывания, процесс разделен на 5 скриптов. Рекомендуется запускать их последовательно из корня проекта.

##### Шаг 1: Подготовка рабочего пространства

Скрипт создает структуру директорий для данных и логов.

```bash
mkdir -p data/raw_synthea
mkdir -p data/raw_oncology
mkdir -p data/fhir_export
mkdir -p toolse/hapi-fhir-jpaserver-starter
```

##### Шаг 2: Получение исходного кода сервера

Загрузка актуальной версии HAPI FHIR JPA Starter с GitHub.

```bash
# Клонирование репозитория HAPI FHIR
git clone https://github.com/hapifhir/hapi-fhir-jpaserver-starter.git \
    toolse/hapi-fhir-jpaserver-starter
```

##### Шаг 3: Запуск контейнеров

Запуск связки PostgreSQL + HAPI FHIR + ClickHouse использованием конфигурации из `.env`.

```bash
cd infrastructure
# Используем docker-compose для поднятия всей связки
docker compose up -d
```

##### Шаг 4: Верификация и проверка работы сервера

Проверка статуса контейнеров и доступности FHIR API.

```bash
# Проверка статуса системы
docker ps --filter "name=hapi"
# Проверка логов сервера (последние 5 строк)
docker logs --tail 5 hapi-fhir-server
# 1. Проверка доступности FHIR API
curl -i http://localhost:8080/fhir/metadata
# 2. Проверка счетчика пациентов (после загрузки)
curl http://localhost:8080/fhir/Patient?_summary=count
```

### Раздел 3. Формирование когорты и загрузка данных (ETL)

Для обучения прогностической модели была реализована стратегия «Золотой середины»: из общей сгенерированной популяции в **78 203** пациента была отобрана сбалансированная выборка, обеспечивающая высокую статистическую мощность при сохранении производительности системы.

#### 3.1 Состав загружаемого датасета:
*   **Целевая группа:** 5 673 пациента с подтвержденным диагнозом рака толстой кишки (коды SNOMED-CT: 363406005, 109838007, 93761005, 254637007, 254632001).
*   **Контрольная группа:** 5 000 случайных «здоровых» пациентов для обучения классификаторов.
*   **Инфраструктура:** 100% сгенерированных медицинских организаций (Organization), локаций (Location) и врачей (Practitioner) для обеспечения ссылочной целостности (Referential Integrity).

#### 3.2 Техническая реализация загрузки (Uploader):
Загрузка выполняется разработанным Python-скриптом `src/fhir_uploader.py`, который реализует:
*   **Многопоточность (Concurrency):** Использование `ThreadPoolExecutor` (20 рабочих потоков) позволило сократить время загрузки 11 000 полных историй болезни до ~40 минут.
*   **Обеспечение целостности:** В отличие от первоначального плана, административные ссылки (на врачей и организации) **сохранены**. Это позволяет проводить анализ влияния квалификации персонала и ресурсов клиники на исходы лечения.
*   **Метод Upsert (PUT):** Ресурсы загружаются с сохранением оригинальных UUID из Synthea, что предотвращает дублирование данных при повторных запусках.

```bash
# Запуск многопоточного загрузчика
python3 src/fhir_uploader.py
```

### Раздел 4. Реализация Bulk Export Pipeline

Извлечение данных для аналитики реализовано через протокол **FHIR Bulk Data Access ($export)**. Это позволяет минимизировать нагрузку на транзакционную базу данных и получить данные в формате, оптимизированном для Big Data.

#### 4.1 Особенности реализации в проекте:
*   **Асинхронность:** Процесс инициируется запросом с заголовком `Prefer: respond-async`.
*   **Хранение Binary:** Сервер HAPI FHIR формирует NDJSON-файлы и сохраняет их во внутреннем хранилище (Binary ресурсы).
*   **Оркестрация экспорта:** Скрипт `src/bulk_exporter.py` выполняет полный цикл: инициирование -> мониторинг статуса (polling) -> скачивание сформированных файлов.

#### 4.2 Формат NDJSON и его аналитическая ценность:
Для Главы 1.5 и последующей загрузки в ClickHouse выбран формат **NDJSON (Newline Delimited JSON)**, так как:
*   **Потоковая обработка:** Каждая строка — это отдельный ресурс. Это позволяет ClickHouse импортировать данные параллельно, используя все ядра CPU.
*   **Масштабируемость:** Формат позволяет обрабатывать миллионы записей без переполнения оперативной памяти (RAM), что критично для медицинских данных.

#### 4.3 Команды управления экспортом:

```bash
# Полный цикл: запуск экспорта и автоматическое скачивание файлов
python3 src/bulk_exporter.py

# Валидация выгруженных файлов в локальной директории
ls -lh data/fhir_export/
# Подсчет общего количества ресурсов для анализа (должно совпадать с загруженным)
wc -l data/fhir_export/*.ndjson
```

### Раздел 5. Аналитическое хранилище и ETL-пайплайн (ClickHouse)

Заключительным этапом создания инфраструктуры (Глава 1.5) является трансформация иерархических FHIR-данных в плоскую колоночную структуру, оптимизированную для машинного обучения и статистического анализа.

#### 5.1 Архитектура ETL-процесса
Для загрузки данных реализован модульный подход **«Один загрузчик — много парсеров»** (скрипт `src/master_loader.py`). Это позволяет масштабировать систему на новые ресурсы (Observation, Procedure) без дублирования кода.

**Ключевые особенности реализации:**
*   **Трансформация данных:** Конвертация JSON-строк в нативные объекты Python (`datetime`, `uuid`), что обеспечивает корректную бинарную вставку в ClickHouse.
*   **Типизация "Медицинского класса":** 
    *   Использование **`Date32`** вместо стандартного `Date` для поддержки истории пациентов, родившихся до 1970 года (критично для онкологии).
    *   Использование **`Nullable`** полей для обработки разреженных медицинских данных (например, даты смерти только у умерших).
    *   Оптимизация хранения через **`LowCardinality`** (для Расы, Пола) и **`Enum8`** (для BirthSex).
*   **Отказоустойчивость:** Пакетная загрузка (batching) через кортежи (tuples), что исключает ошибки сопоставления колонок и обеспечивает максимальную скорость импорта.

#### 5.2 Инициализация схем данных
Перед загрузкой необходимо создать целевую базу данных и таблицы. Схемы спроектированы с учетом специфики прогнозирования рака толстой кишки.

```sql
-- Создание базы данных patients
CREATE DATABASE IF NOT EXISTS oncology;

-- Таблица пациентов (фрагмент схемы)
CREATE TABLE IF NOT EXISTS oncology.patients (
    patient_id UUID,
    birth_sex Enum8('M' = 1, 'F' = 2, 'UNK' = 3),
    birth_date Nullable(Date32),
    deceased_at Nullable(DateTime64(3, 'UTC')),
    is_deceased UInt8,
    race LowCardinality(String),
    -- ... остальные поля ...
    updated_at DateTime DEFAULT now()
) ENGINE = MergeTree() ORDER BY (patient_id);
```

#### 5.3 Команды запуска загрузки

```bash
# Запуск универсального загрузчика
python3 src/master_loader.py

# Верификация загрузки в ClickHouse
clickhouse-client --database oncology --query "SELECT count() FROM patients"
```
